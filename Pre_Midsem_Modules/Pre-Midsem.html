End-to-End Pipeline Flowchart
Raw telemetry enters at Module 1 and flows through validation, state tracking, safety gating, parallel context analysis, dual optimization, and savings quantification before a human-readable recommendation is generated. The Safety Engine (M4) is the mandatory gatekeeper — no optimization can proceed if it returns BLOCKED.

Complete Pipeline Pseudocode
Step-by-step representation of how all 11 modules interact. Each module's inputs, decision logic, and outputs are shown inline. This pseudocode follows Python conventions but is written for readability, not execution.

═══════════════════════════════════════════════════════════════════ ENERVIA OPTIMIZATION PIPELINE — COMPLETE PSEUDOCODE Phase-1 | Read-Only | Advisory Only | No Control Commands ═══════════════════════════════════════════════════════════════════ INPUT : raw_payload ← JSON or CSV string arriving from EEG-01 edge gateway OUTPUT: final_rec ← FinalRecommendationRecord (action, window, savings, safety, reason) FUNCTION run_enervia_pipeline(raw_payload): ── MODULE 1: DATA INGESTION ──────────────────────────────────────── record ← EnerviaIngestion.process_incoming_payload(raw_payload) // Auto-detects JSON vs CSV, standardizes timestamp, normalizes all units. // // OUTPUT — IngestedTelemetryRecord: // timestamp → datetime ISO 8601 standardized // rack_temps → List[°C] always wrapped as a list (even single value) // airflow_proxy → float fan speed % or RPM // chiller_power → float kW // facility_power → float kW // ups_load → float kW or % // tariff → float ₹/kWh // raw_format → str 'json' | 'csv' audit.log(event=INPUT_RECEIVED, source="enervia_ingestion", data=record) ── MODULE 2: DATA VALIDATION ─────────────────────────────────────── validated ← EnerviaValidation(previous_record=last_record).validate_record(record) // Range checks: max(rack_temps) ≤ 45°C | all power values ≥ 0 // Stale check: timestamp present + power values not frozen from last reading // Spike check: |Δfacility_power| ≤ 500 kW vs previous sample // // OUTPUT — ValidatedTelemetryRecord: // data → IngestedTelemetryRecord (original data, unchanged) // is_valid → bool KEY GATE SIGNAL // validation_flags → List[str] e.g. "CRITICAL_TEMP_EXCEEDED" // reason_codes → List[str] human-readable details audit.log(event=VALIDATION_RESULT, data=validated) IF validated.is_valid == FALSE: RETURN "PIPELINE HALTED — invalid telemetry. No recommendation generated." ── MODULE 3: STATE STORE ─────────────────────────────────────────── state ← EnerviaStateStore.update_state(validated) // Maintains 3 rolling buffers: // buffer_15m → deque(maxlen=15) ~15-minute window at 1-min sampling // buffer_1h → deque(maxlen=60) ~1-hour window // buffer_24h → deque(maxlen=1440) ~24-hour window // // Derived metrics computed from buffers: // delta_t_trend = (T_now − T_earliest) / elapsed_minutes // avg_rack_temp_15m = mean(rack_temps) across buffer_15m // avg_facility_power_1h = mean(facility_power) across buffer_1h // is_stable = (delta_t_trend < 0.5°C/min) // last_safe_data = last record where is_valid was True // // OUTPUT — SystemStateRecord: // current_data → ValidatedTelemetryRecord // last_safe_data → IngestedTelemetryRecord | None // delta_t_trend → float (°C/min) // avg_rack_temp_15m → float (°C) // avg_facility_power_1h → float (kW) // is_stable → bool // window_15m_count → int (how many points are in buffer) ── MODULE 4: SAFETY ENGINE — GLOBAL GATEKEEPER ───────────────────── safety ← EnerviaSafetyEngine.evaluate_safety(state) // ALL four conditions must pass to return SAFE: // // CHECK 1 → validated.is_valid == True → else: INVALID_DATA_STREAM // CHECK 2 → max(rack_temps) < 45.0°C → else: THERMAL_LIMIT_EXCEEDED // CHECK 3 → delta_t_trend < 0.5°C/min → else: RAPID_TEMP_RISE // CHECK 4 → window_15m_count ≥ 5 data points → else: INSUFFICIENT_HISTORY // // OUTPUT — SafetyStatusRecord: // status → SafetyStatus "SAFE" | "BLOCKED" // reason_codes → List[str] which checks failed // description → str full human-readable explanation IF safety.status == BLOCKED: audit.log(event=SAFETY_VETO, status=BLOCKED, data=safety) RETURN FinalRecommendationRecord( action = "Maintain current settings (No Action)", window = current_time_window, savings = "0 kW | ₹0", safety = "BLOCKED", reason = "Safety Veto: " + safety.reason_codes ) ✓ SAFETY CLEARED — OPTIMIZATION MAY PROCEED ✓ ── MODULE 5: TARIFF & TIME CONTEXT ──────────────────────────────── cost ← EnerviaCostContext.evaluate_cost_context( current_time = state.current_data.data.timestamp, current_tariff = state.current_data.data.tariff ) // Time-of-Use (ToU) classification (configurable windows): // PEAK: 10:00–14:00 and 18:00–22:00 // OFF-PEAK: 00:00–06:00 // NORMAL: all other hours // Override: IF tariff ≥ 1.8 × base_tariff(6.0) → always classify as PEAK // // Cost severity index (1–10): // Base: OFF-PEAK → 1 | NORMAL → 5 | PEAK → 8 // +2 if tariff > 2.0 × base | +1 if tariff > 1.2 × base (capped at 10) // // OUTPUT — CostContextRecord: // timestamp → datetime // current_tariff → float (₹/kWh) // tariff_status → TariffStatus "PEAK" | "NORMAL" | "OFF-PEAK" // cost_severity_index → int (1–10) ← injected into M7 and M8 ── MODULE 6: OVERCOOLING DETECTION ──────────────────────────────── thermal ← ThermalMetrics( rack_inlet_temps = state.current_data.data.rack_temps, rack_outlet_temps = state.outlet_temps, supply_air_temp = state.supply_temp, return_air_temp = state.return_temp ) overcooling ← EnerviaOvercoolingDetector.evaluate_overcooling(thermal) // Calculations: // mean_delta_t = mean(outlet_temps) − mean(inlet_temps) // thermal_headroom = 27.0°C − max(inlet_temps) [target, not hard limit] // // Overcooling score (0.0 = efficient → 1.0 = highly wasteful): // temp_factor = max(0, (27 − max_inlet) / 10) [weight: 70%] // delta_factor = max(0, (5 − delta_t) / 5) if delta<5 [weight: 30%] // score = min(1.0, 0.7 × temp_factor + 0.3 × delta_factor) // // Feasibility hint: True IF (headroom > 2.0°C) AND (mean_delta_t > 2.0°C) // // OUTPUT — OvercoolingAssessmentRecord: // overcooling_score → float (0.0 – 1.0) // reduction_feasibility_hint → bool ← used by M7 as gate // mean_delta_t → float (°C) // thermal_headroom → float (°C) ── MODULE 7: COOLING OPTIMIZER ──────────────────────────────────── cooling_rec ← EnerviaCoolingOptimizer.generate_cooling_recommendation( safety = safety, overcooling = overcooling, current_power = state.current_data.data.chiller_power, cost = cost ) // Decision tree: // IF safety.BLOCKED → reduction = 0%, safety_status = "BLOCKED" // IF NOT feasibility_hint → reduction = 0%, safety_status = "SAFE" (no waste found) // ELSE: // base_reduction = overcooling_score × 10.0 → [0.0 – 10.0 %] // economic_weight = cost_severity_index / 10.0 → [0.1 – 1.0] // final_pct = min(base × weight, 15.0) → hard cap at 15% // // OUTPUT — CoolingRecommendationRecord: // recommended_reduction_percentage → float (% of chiller load to reduce) // time_window → str "HH:MM–HH:MM" // safety_status → str "SAFE" | "BLOCKED" // explanation → str rationale for the decision ── MODULE 8: PEAK DEMAND OPTIMIZER ──────────────────────────────── peak_rec ← EnerviaPeakOptimizer.evaluate_peak_demand( facility_power = state.current_data.data.facility_power, ups_load = state.current_data.data.ups_load, cost_context = cost ) // UPS safety gate: IF ups_load > 85% → NONE_SAFETY_LOCKOUT (block mitigation) // // Activation condition (either triggers): // CONDITION A: cost.tariff_status == PEAK // CONDITION B: facility_power > FACILITY_PEAK_LIMIT × 0.9 // // Strategy selection: // IF cost_severity_index ≥ 8 → "AGGRESSIVE_LOAD_REDUCTION" // ELSE → "MODERATE_PEAK_SHAVE" // target_kw = facility_power × 0.05 (5% of live facility power) // // OUTPUT — PeakMitigationRecord: // is_mitigation_active → bool // mitigation_strategy → str "AGGRESSIVE_LOAD_REDUCTION" | "MODERATE_PEAK_SHAVE" // | "STANDARD_OPERATIONS" | "NONE_SAFETY_LOCKOUT" // peak_shaving_target_kw → float (kW) // ups_safety_buffer → float (% remaining UPS capacity) // urgency_score → int (1–10) ── MODULE 9: SAVINGS ESTIMATION ─────────────────────────────────── savings ← EnerviaSavingsEngine.compute_savings( baseline_power = state.current_data.data.chiller_power, reduced_power = baseline × (1 − cooling_rec.recommended_reduction_percentage / 100), tariff = state.current_data.data.tariff, duration_hours = 2.0 ← default 2-hour optimization window ) // Formula: // kw_saved = max(0, baseline_power − reduced_power) // energy_saved_kwh = kw_saved × duration_hours // rupees_saved = energy_saved_kwh × tariff // // State tracking (cumulative): // cumulative_kw_saved += kw_saved // cumulative_rupees_saved += rupees_saved // // OUTPUT — SavingsEstimationRecord: // kw_saved → float instantaneous reduction // energy_saved_kwh → float over duration window // rupees_saved_per_interval → float ₹ for this recommendation window // timestamp → datetime ── MODULE 10: RECOMMENDATION ENGINE ─────────────────────────────── // Build context_reasons for explainability reasons ← [] IF overcooling.overcooling_score > 0.3 : reasons.append("Overcooling detected") IF cost.tariff_status == PEAK : reasons.append("Peak tariff window") IF peak_rec.is_mitigation_active : reasons.append("Peak demand mitigation active") final_rec ← EnerviaRecommendationEngine.generate_final_recommendation( recommendation = cooling_rec, savings = savings, safety_status = safety, context_reasons = reasons ) // Decision: // IF safety BLOCKED → action = "Maintain current settings" savings = "0 kW | ₹0" // IF reduction_pct ≤ 0 → action = "System at peak efficiency" savings = "0 kW | ₹0" // ELSE → action = "Reduce cooling by X%" // // OUTPUT — FinalRecommendationRecord: // action → str "Reduce cooling by 7%" // window → str "18:00–20:00" // savings → str "120 kW | ₹1,800" // safety → str "SAFE" | "BLOCKED" // reason → str "Overcooling detected + Peak tariff window" ── MODULE 11: AUDIT LOG (runs throughout entire pipeline) ────────── audit.log_event( event_type = RECOMMENDATION_GENERATED, data_record = final_rec, module_source = "enervia_recommendation_engine", status = SUCCESS ) // AuditLogEntry structure (JSON output): // { // "timestamp" : "2024-01-15T18:00:00Z", // "event_type" : "RECOMMENDATION_GENERATED", // "status" : "SUCCESS", // "payload" : { ...serialized final_rec... }, // "module_source": "enervia_recommendation_engine" // } RETURN final_rec ═══════════════════════════════════════════════════════════════════ EXAMPLE OUTPUT ═══════════════════════════════════════════════════════════════════ action : "Reduce cooling by 7%" window : "18:00–20:00" savings : "120 kW | ₹1,800" safety : "SAFE" reason : "Overcooling detected + Peak tariff window" ═══════════════════════════════════════════════════════════════════ END FUNCTION
Module Reference — Variables & Descriptions
Each module is self-contained and communicates through strongly-typed dataclass records. Inputs, outputs, and key constants are listed for every module. Color coding matches the pipeline flowchart.
M1
Data Ingestion Service
class EnerviaIngestion  |  enervia_ingestion
Receives raw JSON or CSV streams from the edge gateway, auto-detects format, parses timestamps, normalizes all units, and emits a clean, typed telemetry record.
Inputs
payloadstrRaw JSON or CSV string from EEG-01
Constants / Config
——Stateless. No thresholds. Pure parsing.
Output — IngestedTelemetryRecord
timestampdatetimeISO 8601 standardized
rack_tempsList[float]Rack inlet temps in °C (always a list)
airflow_proxyfloatFan speed % or RPM
chiller_powerfloatChiller load in kW
facility_powerfloatTotal facility load in kW
ups_loadfloatUPS load in kW or %
tarifffloatCurrent electricity cost ₹/kWh
raw_formatstr'json' or 'csv' (source metadata)
M2
Data Validation & Sanity Checker
class EnerviaValidation  |  enervia_validation
Enforces physical constraints — catches impossible temperature values, negative power readings, stale sensor data, and sudden anomalous spikes before any optimization logic sees the data.
Inputs
recordIngested TelemetryRecordOutput from M1
previous_recordOptionalLast record, for stale/spike checks
Key Thresholds
MAX_RACK_INLET_TEMPfloat45.0 °C hard limit
SPIKE_THRESHOLD_POWERfloat500 kW max jump per sample

Validation Flags
CRITICAL_TEMP_EXCEEDED—Any rack temp > 45°C
NEGATIVE_POWER_*—chiller / facility / ups below zero
MISSING_TIMESTAMP—No timestamp present
STALE_DATA_WARNING—Power readings frozen from last sample
UNREALISTIC_SPIKE_DETECTED—Δfacility_power > 500 kW
Output — ValidatedTelemetryRecord
dataIngested TelemetryRecordOriginal M1 record (unchanged)
is_validboolKEY GATE: False halts pipeline
validation_flagsList[str]All triggered flag names
reason_codesList[str]Human-readable details per flag

M4
Safety Engine — Global Gatekeeper ⚠
class EnerviaSafetyEngine  |  enervia_safety_engine
The system-wide veto authority. No optimization module can proceed without SAFE approval from this engine. It cannot be bypassed, overridden, or disabled. This is non-negotiable IP.
Inputs
stateSystemStateRecordOutput from M3

Hard Constraints
TEMP_LIMIT_MAXfloat45.0 °C absolute ceiling
MAX_ALLOWED_TRENDfloat0.5 °C/min max rise rate
MIN_REQUIRED_HISTORYint5 points min buffer before decisions
Safety Checks (all must pass)
CHECK 1—validated.is_valid == True → else: INVALID_DATA_STREAM
CHECK 2—max(rack_temps) < 45°C → else: THERMAL_LIMIT_EXCEEDED
CHECK 3—delta_t_trend < 0.5°C/min → else: RAPID_TEMP_RISE
CHECK 4—window_15m_count ≥ 5 → else: INSUFFICIENT_HISTORY
Output — SafetyStatusRecord
statusSafetyStatusSAFE or BLOCKED
reason_codesList[str]Which checks failed (empty if SAFE)
descriptionstrFull human-readable explanation
M5
Tariff & Time Context Engine
class EnerviaCostContext  |  enervia_cost_context
Translates the raw electricity price and current time into an economic priority index. Tells both M7 and M8 how aggressively to optimize — "energy saved" and "money saved" are different things.
Inputs
current_timedatetimeFrom state record timestamp
current_tarifffloat ₹/kWhLive electricity price
Configuration
base_tarifffloat₹6.0/kWh baseline reference
peak_windowsList[tuple]10:00–14:00, 18:00–22:00
off_peak_windowsList[tuple]00:00–06:00
severity scaleint 1–10OFF-PEAK→1 | NORMAL→5 | PEAK→8 (+ price modifier)
Output — CostContextRecord
timestampdatetimeCurrent time
current_tarifffloat₹/kWh as provided
tariff_statusTariffStatusPEAK NORMAL OFF-PEAK
cost_severity_indexint 1–10Economic urgency scale
M6
Overcooling Detection Engine
class EnerviaOvercoolingDetector  |  enervia_overcooling_detector
Detects and quantifies cooling waste by analyzing temperature differentials across racks. Determines whether the system is provisioning more cooling than the heat load demands, and signals whether a safe reduction is feasible.
Inputs — ThermalMetrics
rack_inlet_tempsList[float] °CInlet temperatures per rack
rack_outlet_tempsList[float] °COutlet temperatures per rack
supply_air_tempfloat °CCold aisle supply temperature
return_air_tempfloat °CHot aisle return temperature
Key Thresholds & Formula
TARGET_INLET_MAXfloat27.0 °C optimization target (not hard limit)
MIN_SAFE_DELTA_Tfloat5.0 °C below = likely airflow waste
score formula—0.7 × temp_factor + 0.3 × delta_factor
feasibility gateboolheadroom > 2°C AND delta_t > 2°C
Output — OvercoolingAssessmentRecord
overcooling_scorefloat 0.0–1.00=efficient, 1=highly wasteful
reduction_feasibility_hintboolKEY INPUT to M7 — safe to reduce?
mean_delta_tfloat °CAvg temp rise across racks
thermal_headroomfloat °CGap between max inlet and 27°C target
M7
Cooling Optimization Engine
class EnerviaCoolingOptimizer  |  enervia_cooling_optimizer
The core decision engine for cooling reduction. Integrates safety approval, overcooling evidence, and economic context to compute the exact percentage reduction that is both safe and cost-effective.
Inputs
safetySafety StatusRecordFrom M4 — MUST be SAFE
overcoolingOvercooling AssessmentRecordFrom M6
current_powerfloat kWLive chiller power (from state)
costCostContextRecordFrom M5 (tariff + severity)
Constants & Formula
MAX_REDUCTION_CAPfloat15.0% hard cap per step
BASE_WINDOW_HOURSint2 hours default window
base_reductionformulaovercooling_score × 10 → 0–10%
economic_weightformulacost_severity_index / 10 → 0.1–1.0
final_pctformulamin(base × weight, 15.0)
Output — CoolingRecommendationRecord
recommended_reduction_percentagefloat %% of chiller load to reduce
time_windowstr"HH:MM–HH:MM" recommendation window
safety_statusstr"SAFE" | "BLOCKED"
explanationstrHuman-readable decision rationale
M8
Peak Demand Optimization Engine
class EnerviaPeakOptimizer  |  enervia_peak_optimizer
Hyper-focused on reducing demand charges during high-cost windows. Coordinates with UPS safety margins (read-only) to ensure peak shaving recommendations do not push backup systems past critical thresholds.
Inputs
facility_powerfloat kWTotal facility load (from state)
ups_loadfloat %UPS load percentage (from state)
cost_contextCostContextRecordFrom M5 (tariff + severity)
Constants & Logic
FACILITY_PEAK_LIMITfloat kW1000 kW configurable per site
UPS_CRITICAL_LOAD_THRESHOLDfloat %85% → lockout if exceeded
activation condition—PEAK window OR >90% of peak limit
target_kw formula—facility_power × 0.05 (5% reduction target)
Output — PeakMitigationRecord
is_mitigation_activeboolWhether to recommend mitigation
mitigation_strategystrAGGRESSIVE_LOAD_REDUCTIONMODERATE_PEAK_SHAVESTANDARD_OPERATIONSNONE_SAFETY_LOCKOUT
peak_shaving_target_kwfloat kWTarget kW to shave
ups_safety_bufferfloat %100 − ups_load (remaining capacity)
urgency_scoreint 1–10Maps from cost_severity_index
M9
Savings Estimation Engine
class EnerviaSavingsEngine  |  enervia_savings_engine
Converts physical power reductions into quantified financial impact. Provides the "credibility numbers" — kW and ₹ savings — that make recommendations actionable and verifiable.
Inputs

reduced_powerfloat kWPower with optimization applied
tarifffloat ₹/kWhCurrent electricity cost
duration_hoursfloatLength of optimization window (default: 2h)
Formulas
kw_savedformulamax(0, baseline − reduced)
energy_saved_kwhformulakw_saved × duration_hours
rupees_savedformulaenergy_saved_kwh × tariff

Internal State
cumulative_kw_savedfloatRunning total across all calls
cumulative_rupees_savedfloatRunning total ₹ saved
Output — SavingsEstimationRecord
kw_savedfloat kWInstantaneous power reduction
energy_saved_kwhfloat kWhEnergy saved over window duration
rupees_saved_per_intervalfloat ₹Financial savings for this window

M10
Recommendation Generator
class EnerviaRecommendationEngine  |  enervia_recommendation_engine
The final synthesizer. Assembles all upstream outputs into a clear, explainable, human-readable advisory recommendation. Operators trust explainable actions, not black boxes.
Inputs
recommendationCooling Recommendation RecordFrom M7
savingsSavings Estimation RecordFrom M9
safety_statusSafety Status RecordFrom M4
context_reasonsList[str]Human-readable trigger list
Decision Logic
if BLOCKED—"Maintain current settings" | savings="0 kW | ₹0"
if reduction ≤ 0—"System at peak efficiency"
else—"Reduce cooling by X%" with full savings
Output — FinalRecommendationRecord
actionstre.g. "Reduce cooling by 7%"
windowstre.g. "18:00–20:00"
savingsstre.g. "120 kW | ₹1,800"
safetystrSAFE or BLOCKED
reasonstre.g. "Overcooling detected + Peak tariff window"
M11
Audit & Logging Service
class EnerviaAuditLog  |  enervia_audit_log
Provides a structured, immutable audit trail for every pipeline event. Called from all modules — never blocks execution. Critical infrastructure requires full traceability.
Event Types Logged
INPUT_RECEIVEDfrom M1Raw telemetry arrived
VALIDATION_RESULTfrom M2Pass or fail + flags
SAFETY_VETOfrom M4BLOCKED decision + reason codes
OPTIMIZATION_COMPUTEDfrom M7/M8Optimizer decision result
RECOMMENDATION_GENERATEDfrom M10Final advisory output
SYSTEM_ERRORanyException caught in pipeline
Audit Status Values
SUCCESS—Normal execution
WARNING—is_valid==False or stale data
BLOCKED—Safety veto applied
CRITICAL—System error / logging failure
Output — AuditLogEntry (JSON)
timestampstrISO 8601 UTC timestamp
event_typestrFrom EventType enum
statusstrSUCCESS | WARNING | BLOCKED | CRITICAL
payloadDictSerialized dataclass (via asdict())
module_sourcestrWhich module generated the event
